# 13 开发约束与扩展指南

## 13.1 开发约束（工程红线）

以下约束用于避免系统长期演进中的架构退化：

1. 不做只修局部的补丁式改法，优先考虑全链路一致性。
2. 不引入过多中间态，避免可读性下降与循环依赖。
3. 不用大量过渡防御代码掩盖主逻辑。
4. 不以“功能可用”替代架构质量。
5. 关键模块必须有解释性注释，尤其是跨层编排逻辑。
6. 保持函数职责单一，避免超长函数和多语义混杂。
7. 严格遵循 SOLID 与 DRY。
8. UI 状态必须从核心数据推导，不依赖临时状态驱动。
9. 状态变化必须显式，避免隐式时序耦合。
10. 不吞异常，不允许空 catch。
11. 错误语义必须统一，不混用 null/false/undefined 风格信号。
12. 不在多处维护同一事实。
13. 不在核心业务逻辑中混入直接 IO/外部副作用。
14. 不新增接口前先评估是否可复用现有能力。
15. 改动后必须补测试并执行验证。

## 13.2 分层依赖规则

允许依赖：

- `api -> services`
- `services -> repositories/policy/gateway/analyzer`
- `repositories -> models`

禁止依赖：

- `api -> repositories` 直接写库。
- `policy -> api` 反向依赖。
- `analyzer -> ui` 反向依赖。

## 13.3 扩展点说明

### 新增工具类型

步骤：

1. 在 `app/gateway/action_intent.py` 增加动作识别与参数规范化逻辑。
2. 新建 interceptor 提取资源身份。
3. 新建 executor 执行逻辑。
4. 在 `gateway_manager` 注册。
5. 新增规则与测试。

### 新增策略规则

步骤：

1. 更新 `configs/rules` 对应 YAML。
2. 如新增规则类别，扩展 loader + matcher + models。
3. 保证 explain 输出可读。
4. 补单元与集成测试。

### 新增风险链

步骤：

1. 在 `risk_analyzer` 增加检测函数。
2. 输出 `RiskFinding`（含 chain_id/path_nodes）。
3. 确保 graph/report 能展示。
4. 补图谱与报告测试。

### 新增 API 接口

步骤：

1. 先确认现有接口是否可复用。
2. 新接口遵循统一响应与错误协议。
3. API 层只做编排入口，不写复杂业务逻辑。
4. 添加最小集成测试。

## 13.4 开发自检清单

提交前建议逐项确认：

- 是否引入了不必要的新状态。
- 是否破坏了分层边界。
- 是否复用了统一错误协议。
- 是否更新了 `docs/manual` 对应章节。
- 是否补齐了回归测试。

## 13.5 文档更新策略

当发生下面任一变化，必须更新说明书：

- API 路径或字段变更。
- 规则结构变更。
- 风险链判定逻辑变更。
- 运行或排障步骤变更。
